# 安全模块化发布工程

[English](README.md)

这是一套面向 AI 编程代理的证据驱动型工程指令，结合了可维护软件架构、严谨实施、生产上线验收与防御性安全审查。

该 Skill 帮助 AI 编程代理构建模块清晰、可测试、可审查、具备运维准备并且更安全的软件。支持 Codex、Claude Code、Cursor、Trae、Windsurf、GitHub Copilot，以及其他支持可复用 Skill、项目规则或自定义指令的工具。

## 强制执行内容

- 建立清晰的业务模块与依赖边界。
- 每个文件只有一个主要职责和一个主要变更原因。
- 手写源代码文件不得超过 800 个物理行。
- 在提出实现结论或架构决策前获取仓库证据。
- 编写代码前调用相关流程与技术 Skill。
- 完成任务前运行相关测试、验证命令并提供证据。
- Bug 无法解决时停止盲目尝试，主动研究并及时告知用户。
- 使用明确证据执行生产上线验收。
- 执行防御性、已授权的安全测试和攻击者视角威胁审查。

## 适用场景

在以下场景使用本 Skill：

- 实现功能或业务流程；
- 修复 Bug 或调查异常行为；
- 重构大型、职责混合或强耦合代码；
- 审查架构、模块归属或依赖方向；
- 准备预发布或生产环境构建；
- 执行上线准备、安全或渗透测试审查；
- 验证用户、管理员、租户或服务账号权限；
- 设计认证、密码恢复、注册或滥用防护机制。

## 开发原则

### 模块职责

每项职责都必须具有明确归属。没有充分理由时，不得将 UI、编排、状态、数据访问、领域逻辑、类型、常量、格式化和配置混合在一起。

业务模块只公开有意设计的公共契约。其他模块不得依赖其私有实现文件。

### 文件职责与大小

每个手写源代码文件应只有一个主要职责和一个主要变更原因。

硬性上限为 800 个物理行。文件达到约 250 至 350 行时审查职责，达到约 600 行时主动规划拆分。已有超限文件必须先拆分，才能继续添加功能。

### 证据优先实施

不得虚构文件、API、字段、命令、依赖、需求、行为或测试结果。

使用仓库代码、契约、Schema、配置、运行时输出、测试或官方文档验证决策。无法验证的假设必须明确标注。

## 无法解决的 Bug

禁止长时间重复尝试相同的失败方案。

经过少量基于证据的本地尝试后：

1. 停止反复试错。
2. 查询精确错误信息、官方文档、上游仓库、发布说明和权威技术资料。
3. 确认资料适用于项目的实际版本与环境。
4. 安全验证可信方案。
5. Bug 仍无法解决时，及时告知用户。

最终报告必须说明观察到的行为、证据、已尝试方案、研究来源、可能原因、剩余阻塞点和最小后续行动。

## 生产上线验收

上线前，每项适用检查必须记录为：

- `通过`
- `失败`
- `阻塞`
- `不适用`，并提供书面原因

跳过、未知和未测试不能视为通过。

上线验收覆盖：

- 关键用户流程和验收标准；
- 角色、权限、租户隔离与负向路径；
- 输入验证、错误、重试、恢复与兼容性；
- 单元、集成、端到端、回归与静态验证；
- 性能、并发、可靠性与资源使用；
- 部署、配置、迁移、功能开关与回滚；
- 日志、指标、追踪、告警、备份、恢复与事故响应；
- 安全发现、修复、复测与剩余风险。

最终建议必须为 `允许上线`、`有条件上线` 或 `禁止上线`，并由证据支持。

## 用户端与管理端边界

上线前，必须清点所有 UI 操作、API 端点、RPC 方法、任务、Webhook、导入、导出和管理操作。

每项能力必须记录：

- 允许的用户和角色；
- 允许访问的数据与租户范围；
- 认证和验证要求；
- 服务端授权策略；
- 敏感影响和必需审计事件；
- 滥用防护与负向测试证据。

普通用户不得调用管理 API、修改可信字段、提升权限、跨越租户边界、访问无关数据、关闭安全控制，或执行维护与破坏性操作。

管理员能力必须采用最小权限、强 MFA、近期二次认证、审计记录、权限撤销，以及高风险操作审批机制。

客户端可见性、隐藏路由、禁用按钮和 UI 角色检查绝不是授权控制。

## 密码、凭证与账号

本 Skill 要求安全处理密码和凭证：

- 支持长密码短语和密码管理器；
- 拒绝已泄露的密码；
- 禁止静默截断密码；
- 只使用现代、加盐、自适应密码哈希存储密码；
- 禁止保存明文或可逆加密密码；
- 防止账号枚举；
- 限制并检测暴力破解、密码喷洒和撞库攻击；
- 使用短期、单次令牌保护密码重置和账号恢复；
- 对管理员和敏感操作要求 MFA 或二次认证；
- 轮换并撤销会话、Token、API Key 和已泄露凭证；
- 禁止将密钥写入源代码、客户端、日志、构建产物和支持工具。

## 机器人注册与滥用防护

自动化注册必须采用分层防护，而不是仅依赖一个验证码。

防护措施包括：

- 基于账号、设备、网络、行为和目标地址的服务端限流；
- 基于风险逐步增加验证要求；
- 验证流程与单次验证码保护；
- 防止邮件、短信和通知轰炸；
- 为新账号设置保守权限和配额；
- 保护试用、奖励、邀请、消息、投票、上传和 API；
- 防止重放，并确保幂等性和并发安全；
- 监控账号农场和相关滥用账号；
- 支持调查、撤销、申诉和误判统计。

如果简单自动化能够批量获取账号、奖励、配额、消息能力或高价值权限，则必须阻止上线。

## 授权安全测试

安全测试必须是防御性的、已授权的、范围明确的，并由证据驱动。

开始主动测试前必须记录：

- 书面授权和责任人；
- 明确的测试范围与排除资产；
- 获准使用的账号、环境、日期和技术；
- 请求速率与安全限制；
- 监控和事故响应联系人；
- 证据处理与删除规则；
- 回滚和恢复流程。

如果测试越界、影响可用性、暴露无关数据、存在破坏风险，或失去监控与事故响应保障，必须立即停止。

本 Skill 不授权攻击无关系统。系统可公开访问不代表获得测试许可。

## 防御性攻击者视角

针对每项重要功能和信任边界，思考：

1. 攻击者会以哪个资产为目标？
2. 哪个入口可以到达？
3. 哪项假设或控制可能失效？
4. 哪个信任边界可能被跨越？
5. 多个弱点如何组合成攻击路径？
6. 哪些控制负责预防、检测、控制和恢复？
7. 哪些安全证据能够证明控制有效？

审查侦察暴露、身份滥用、权限提升、不安全输入处理、业务逻辑滥用、敏感数据泄露、客户端篡改、供应链风险和事故恢复能力。

根据项目情况使用 OWASP ASVS、WSTG、MASVS、MASTG 和 NIST SP 800-115。

```text
~/.codex/skills/secure-modular-release-engineering
```

### Claude Code

```text
~/.claude/skills/secure-modular-release-engineering
```

在 Claude Code 中仅针对特定仓库使用时，可将适用指令放入项目的 `CLAUDE.md`。

### Cursor、Trae、Windsurf 与 GitHub Copilot

根据工具支持的格式，将 `SKILL.md` 添加为项目规则、代理规则、工作区指令或自定义指令。

保留 `references` 目录，使代理能够按需读取详细的模块化、上线和安全指南。

不同工具和版本的原生 Skill 发现机制及规则文件位置可能不同。

## 上线决策

上线结论分为：

- **允许上线**：关键验收条件通过，并且没有阻止上线的风险。
- **有条件上线**：剩余风险具有明确责任人、期限、监控措施并已正式接受。
- **禁止上线**：关键行为、安全边界、回滚、恢复、授权或证据仍不完整。

没有证据时，绝不能声称软件已经修复、安全、测试完成或可以上线。
